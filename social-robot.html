Intro

A summary of my contribution

To put it simply, let us say the goal here is to give a robot the ability to be trustworthy and easy to connect with emotionally.

As complicated as it is, psychology suggests that most of our perception of others’ feelings is based on what we know as body language and facial expressions. Moreover, children with ASD have a hard time processing sensory data and this might lead to cognitive overload, which explains the oversensitivity to bright lights and loud noises.

Keeping these in mind, I divided the system design procedure into 4 major subsystems:

  • Mechanical system (Movement Module)
  • Affective expression system (cognitive module)
    • Android client
    • Emotion Recognition (A fisheye lens camera )
  • user interface :
    • web back-end

Unifying all above mentioned based on this manifest to help future integrations and developments, thus every module was designed and developed regarding this matter to happen.

System Design

Designing a complex system like social robots is a tedious task that requires multiple interdiciplinary insights on how each and every subsystem works - how to make them all work together as a whole unified system.

My priority and long-term goal was to build a system in which no prior coding would be required to implement extra features in the future aka. clean coding. 

Needless to say, the integration of android, web front, web back and ROS is not so easy to design in a way that it doesn't bring a negative impact to the comunication speed between the gateways. Since this is a research-assistive robot, these subtle errors could lead to a false result.

Mechanical System

The robot designed by the previous team was as shown below. Containing 3 levels of 12mm wooden flat discs and highly concentrated 3d printed heads the overall weight rose above 6 kg. The interaction module is based on an Android tablet that presents the robot’s face. A fisheye lens camera was provided for a wide-angle view. The robot’s brain is an Nvidia jetson nano that controls all Dynamixel motors with a USB adapter. The robot was equipped with a local wifi router for ease of communication which I removed due to several reasons to be discussed later.

 The movement module is based on a triangular Omni wheel vectoring mechanism for the base motion; also, the robot’s hands have one degree of freedom each. Other than that, the head of the robot has 2 degrees of movement provided by a pan-tilt mechanism.

Using two perpendicular aluminium profiles attached by 4 screws to the bottom, the overall weight droped by at least two thirds while saving the stregth needed and also leading to a much less expensive structure. 
These changes also resulted in a more modular body that was easier to changs in the future. (tearing down each section with less effect on the whole system) and less custom-designed parts included. (Best for hypothetical future industrializing.)

Additional to the above-mentioned changes in the skeleton, the loud noise made by the omnidirectional steering module - caused by the gearboxes of A12s - was a huge problem considering the mental overload of children with ASD with loud noises.
So as the only solution available, I replaced the steering module's actuators-dynamixel A 12- with Maxon geared encoder Dc motors, which also required re-design in the movement module.

Software Design

The software is made of : 1-User Interface  2-Web Back-end  3-ROS ecosystem. So, the focus is mainly on the co-operation of these three, while keeping each working independently.( Basicaly the definition of a modular system!) The main reason behind stressing "modular" in every section is the benefits that these kind of systems have, for instance, easier debugging.

Web Back-End

Back-end of the interface is in charge for handling requests made by the user(s) and the android client, which is the emotion module, and sending the right info via compatible responses to them for futher actions. To put it simply, the back is responsible for the first layer of comunication between the user and the action they expect from the robot.

The web interface back-end is programmed using Django REST framework,one of the most powerful web frameworks written in Python. It supports manual settings for both the database (e.c. Postgres, SQL) and the web server used during the production.

In this case, I implemented the system on the local server - Nvidia Jetson - with the combination of Django-Nginx-Gunicorn to be prepared for the load of requests during the research sessions, while being as fast as possible. Since Nginx is best suitable for real website productions, changing a simple line in it's configuration will give the owner the option for non-local comunications as well. As it can be seen, giving each part the ability to be changed and developed easily and faster has been taken into account.
